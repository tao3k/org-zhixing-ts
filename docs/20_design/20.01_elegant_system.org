* 20.01 Elegant System

Org Zhixing's visual system should feel like a modern Org buffer projected into
the browser, not like a marketing dashboard.  The design target combines two
observations:

- =org-modern= treats Org syntax as the interface.  It improves headings,
  labels, keywords, tables, source blocks, tags, and font alignment while
  keeping the underlying Org text editable and meaningful.
- =elegant-emacs= shows that a small set of carefully named faces can carry a
  whole workspace.  The system should be minimal, consistent, and quiet before
  it is decorative.

** Contract

The UI contract is a face economy:

| Face | Purpose |
|------+---------|
| =normal= | default reading text |
| =strong= | headings and primary labels |
| =faded= | secondary text |
| =muted= | counters and structural hints |
| =salient= | navigation and active structure |
| =popout= | warnings or attention |
| =success= | done or valid state |
| =critical= | errors |
| =structural= | code, properties, and schema-like facts |

Every product surface should express itself through this face set, surface
tokens, hairlines, and small radii.  A new color or shadow is a design-system
change, not a page-local flourish.

** TypeScript Package Boundary

The design system is intentionally package-backed:

| Package | Responsibility |
|---------+----------------|
| =@radix-ui/colors= | accessible primitive color scales |
| =@fontsource-variable/inter= | variable-pitch reading face |
| =@fontsource/roboto-mono= | fixed-pitch code and property face |
| =style-dictionary= | compile =tokens/elegant.json= into CSS variables |
| =@capsizecss/core= and =@capsizecss/metrics= | derive type trim variables from font metrics |
| =@zag-js/floating-panel= | window state, drag, resize, maximize, and restore for Zen Glance |
| =masonry-layout= | waterfall layout for mixed media inside Zen Glance |
| =@tanstack/virtual-core= | large Travel list windowing |
| =@mozilla/readability= | future external web-page reader extraction, not the current Org-rendered Zen path |

The point is not to minimize dependencies.  The point is to let mature TS
libraries own the parts that are stateful, measured, or easy to get subtly
wrong.

** Surface Rules

- Reading surfaces are structured by hairlines, indentation, and whitespace
  before cards and shadows.
- Cards are allowed for repeated records, but they should not dominate the
  interface or nest into other cards.
- Labels use =--radius-label=, regular or medium weight, and semantic faces.
  Fully rounded pill shapes are avoided unless the interaction itself is a
  native pill-like control.
- Product text does not use viewport-scaled font sizes.  The system uses token
  sizes and stable line-height contracts.
- The maximum named system weight is =--weight-heavy= at 700.  Stronger visual
  hierarchy should come from spacing, structure, and face choice.
- Zen Glance is a windowed reading surface.  The floating-panel state machine
  owns its size and position; CSS only paints the shell and content.

** Elegant Layout Translation

The layout model is inspired by
[[https://github.com/rougier/elegant-emacs][rougier/elegant-emacs]] and
[[https://github.com/minad/org-modern][minad/org-modern]], but it is translated
into browser constraints instead of copied as Emacs Lisp.

| Reference idea | Org Zhixing rule |
|----------------+------------------|
| Minimal frame chrome | page gutters come from =--frame-inset=; major separation uses hairlines and whitespace |
| 72-column editor frame | rendered Org uses =--reader-width: 72ch= as the default reading measure |
| Window dividers instead of cards | navigation, TOC, and section metadata use =--divider-width= hairlines |
| A few named faces | every component maps back to the face economy instead of local color inventions |
| Line spacing as breathing room | text rhythm comes from tokenized type leading, not viewport-scaled type |
| Org syntax as UI | TODO, priority, timestamp, tag, table, and block atoms participate directly in layout |

Blog is the first explicit frame translation.  The old product layout used a
large library header plus a horizontal article switcher, which felt like a web
dashboard.  The elegant layout now treats the Blog surface as a compact index
space: a header-line, a thin threshold-based time strip, clickable topic
filters, a small association strip, and then the article list as the dominant
content.  The filters must never become a sidebar or a second article wall.
Clicking an article enters a centered 72-column Zen reading body.  Zen hides
site navigation, source controls, and reader-local chrome; it is a pure reading
surface.  =Esc= returns to the index, and arrow keys move to adjacent articles.
Records follow the same direction: source-backed notes are a hairline-separated
index, not a card wall.

SimpRead is a useful product reference because it separates focus mode,
reader-mode extraction, Chinese reading rhythm, media preservation, annotation,
and later-read workflows.  Org Zhixing should translate those ideas into Org
terms instead of copying the browser-extension architecture.  The parser already
owns article structure, so Zen does not need =Readability= for current Org
articles.  It should borrow the reading surface: quiet chrome, centered first
heading, centered article metadata, generous prose rhythm, and preserved media.
If a future capture flow imports arbitrary web pages, =@mozilla/readability=
belongs at that external-ingest boundary together with sanitizer policy, not in
the hot Zen render path.

Large collections must not become large visual grids.  A workspace with 1000 or
2000 Org files still gets a compact source picker, but source files are quiet
substrate rather than the main visual object.  This follows the static-site
mental model used by Hugo: content-backed
[[https://gohugo.io/methods/site/regularpages/][regular pages]] form page
collections, and
[[https://gohugo.io/quick-reference/page-collections/][collection methods]]
can sort those pages by date.  Blog therefore presents an index-first article
collection sorted by article date, with time thresholds, topic facets, and
article titles as the first visual unit.  The time rail must not render one
cell per article.  For 1000 articles it becomes a small set of selectable date
thresholds, while the article list uses the TanStack virtual-core boundary when
the result set is large.  Bulk discovery belongs to search/command surfaces and
virtualized lists, not to the reading frame.

The first viewport should say "life archive" by structure, not by translating
English labels to Chinese.  Navigation labels remain recognizable product
surfaces, while Blog treats source files as quiet substrate.  The visible Blog
unit is an article title with time, topic, and source trace; clicking it enters
Zen reading.

Agent-facing reading is not only a traditional blog index.  Human memory moves
as an association network, while LLM reasoning often proceeds as a branching
tree.  Blog therefore exposes an association lens derived from parser-owned Org
tags and article roots: topic filters model the network entry points, and the
association strip shows compact branches from a root topic to related topics.
It must not repeat article titles or compete with the article index.  This
remains a projection over Org facts; the frontend must not invent another parser
or store agent-written categories as source truth.

The practical test is simple: if a surface needs more elegance, first ask which
Org semantic unit is missing or which frame rhythm is unstable.  Add a card,
shadow, or decorative wrapper only after that answer is clear.

** Org Native Alignment

The frontend should make Org syntax beautiful by consuming parser/exporter
facts, not by reparsing Org text in TypeScript.

| Org-modern area | TypeScript implementation |
|-----------------+---------------------------|
| Headline stars | CSS structural markers on rendered headings |
| TODO and DONE states | =sectionIndex= TODO fields become heading markers with semantic tones |
| Priority cookies | parser priority DTO becomes compact =[#A]/[#B]/[#C]= heading markers |
| Planning timestamps | =OrgizeTimestampDto= becomes =time.org-timestamp= with active/inactive/range state |
| Tags | effective parser tags become compact Org meta labels |
| Property drawers | parser property DTOs become structural =dl= rows, not body text |
| Tables | exported tables are wrapped in =.org-table-frame= and header cells are promoted to semantic =th= |
| Example/source blocks | exported =pre/code= blocks are framed with Org-style block names |
| Checkboxes | only attribute/DTO-backed checkbox state is rendered; if export drops state, the parser must expose it before TS paints it |

This is the boundary: TypeScript can beautify semantic facts, but it must not
invent a second Org parser with regular expressions.

The same visual atoms must be shared across views.  Agenda rows, record cards,
memory cards, Travel cards, attachment cards, and rendered articles should not
each invent a private TODO or planning badge.  The common classes are the
contract: =org-heading-todo=, =org-heading-priority=, =org-planning-chip=,
=org-timestamp=, =org-meta-tag=, and =org-native-table=.

** Tests

The style tests protect the contract:

- =src/styles/theme.css= must be generated from =tokens/elegant.json=.
- every token in =tokens/elegant.json= must appear in the generated theme;
- elegant layout tokens such as =--frame-inset=, =--reader-gutter=,
  =--reader-width=, and =--divider-width= must stay in use by reader surfaces;
- Blog must render an index-first surface by default: dated article titles,
  threshold time navigation, clickable topic filters, and association paths.
  Clicking an article enters Zen reading, where the rendered article is
  constrained by =--reader-width= and no top navigation is visible;
- Zen keyboard controls must stay available: =Esc= exits to the index, and
  arrow keys move through adjacent articles without showing reader chrome;
- source navigation and Blog article indexes must stay bounded for large
  workspaces; tests cover 1000 source files and 20 article headings so the UI
  cannot regress into an unbounded source/file card wall;
- Blog article indexes are ordered as a dated article collection, not by source
  file order or source picker state;
- Blog keeps large article indexes behind a lazy TanStack virtual-core boundary;
- package-backed color and font imports must stay present;
- core reader surfaces must not reintroduce hard-coded hex colors, very heavy
  font weights, =999px= label radii, local card shadows, or hover translation;
- rendered Org remains semantic: headings, TODO/DONE, tags, property rows,
  source blocks, tables, and meta chips are styled by Org meaning, not by page
  cards.
- model render tests verify that TODO, priority, planning timestamps, tables,
  and source/example blocks produce explicit Org-native HTML affordances.

** Current Slice

This slice introduces the generated token contract, migrates the foundation,
blog reader, Travel, attachment gallery, rendered Org, and memory records onto
the face economy, adds explicit Org-native render affordances for headings,
planning timestamps, tables, and blocks in the TypeScript renderer, and starts
sharing those atoms in Agenda rows, record cards, and memory cards.  Checkboxes
stay behind a parser-owned DTO boundary.  Travel and attachment tags now consume
the same Org tag atom instead of private chips.

This follow-up makes the layout change visible: Blog becomes an editor-like
frame with a compact header-line, left rail, and 72-column reading body; Records
become a hairline-separated note index.  Later slices can move the remaining
agenda modules through the same gate once their product layout is revisited.
